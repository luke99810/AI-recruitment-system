## 一、项目背景

| **场景A：AIOffer-Research** | `C:\Users\宿心\WorkBuddy\AI-Interview\AIOffer-Research\` | 简历解析 + 匹配评分 + 试题生成 + Skill模拟面试HTML | FastAPI + 原生HTML | 8000 |

### 1.2 合并目标

将上述两个项目合并为 **统一的 AI 招聘系统**，坐落于 `D:\AI recruitment system\`，使用 **单一 Streamlit 前端** 承载全部功能。两个项目的核心能力完整保留，无缝衔接：

- **简历分析环节**（场景A完整保留）：JD+简历上传 → 结构化解析 → 匹配度评分(0-100) → 12+道5维度面试题 + 模糊点追问 → 环形图+题目卡片展示 → 导出模拟面试HTML
- **AI面试环节**（场景B完整保留）：从上一环节题库中随机抽取N道题 → 面试官角色扮演 → 多轮动态对话(含追问/反思) → TTS播报 → 虚拟数字人
- **评估报告环节**（场景B完整保留）：5维雷达图 + 逐题评审 + 加权总分 + 录用建议 + 矛盾标注

### 3.2 题库抽取策略（新增核心逻辑）

**目标**：分析环节生成的全部题目完整展示给用户看，AI面试从中随机抽取指定数量启动，保留原題的所有元信息。

```python
def sample_questions_for_interview(all_questions: list, n_rounds: int) -> list:
    """
    从完整题库中随机抽取 N 道题用于AI面试。
    
    抽取规则：
    1. N = min(n_rounds, len(all_questions))
    2. 按5维度分层抽样，保证每个维度至少有1题（如果该维度有题的话）
    3. 每道题保留原始元信息：question, tags, hints, answer, difficulty
    4. 剩余题目作为Agent追问/切换时的备选题池
    """
    from random import sample
    
    # 按维度分组
    by_dimension = defaultdict(list)
    for q in all_questions:
        for tag in q.get("tags", []):
            by_dimension[tag].append(q)
    
    selected = []
    # 第一轮：每个维度保底1题
    for dim, qs in by_dimension.items():
        if qs:
            selected.append(sample(qs, 1)[0])
    
    # 第二轮：随机补足到 N
    remaining_pool = [q for q in all_questions if q not in selected]
    needed = n_rounds - len(selected)
    if needed > 0 and remaining_pool:
        extra = sample(remaining_pool, min(needed, len(remaining_pool)))
        selected.extend(extra)
    
    return selected[:n_rounds]
```

**数据流**：
```
简历分析Tab 完成 → all_questions 存入 st.session_state
     ↓
用户点击 [开始AI面试]
     ↓
sample_questions_for_interview(all_questions, n_rounds=8) → 抽取8题
     ↓
存入 st.session_state.interview_questions + st.session_state.remaining_pool
     ↓
跳转 AI面试Tab → interviewer.init_with_questions(selected_questions)
     ↓
Agent按序提问 + 动态追问 + 必要时从remaining_pool补题
```

### 3.7 面试官Agent改造（新增：题库注入）

原场景B的InterviewerAgent在初始化时自行调用LLM生成首轮问题。合并后改造为：

```
原流程: parse_documents() → init_persona() → 【LLM生成首轮问题】
新流程: parse_documents() → init_persona() → inject_questions(sampled_questions) → 面试循环
```

`inject_questions()` 方法：
- 接收从题库抽取的问题列表
- 设置为首轮提问队列
- 保留原題的 `answer`（评分标准）用于评估时的参考答案对比
- `remaining_pool` 中的题作为Agent追问/切换时的备选来源

Agent在面试循环中：
1. 先按注入题库顺序逐题提问
2. 每题回答后执行反思决策
3. 如果决定"追问"：在同一维度生成新追问（动态生成，不消耗题库）
4. 如果决定"切换维度"：从remaining_pool中按目标维度取题
5. 如果题库耗尽且维度已覆盖：结束面试

### 4.1 面试官反思决策机制（场景B核心）

**挑战**：如何让AI面试官不机械地按顺序出题，而是像真人一样动态决策？

**方案**：每轮面试后执行"评估 → 反思 → 决策"三元循环

```
评估:  候选人回答质量打分 (1-5)，识别矛盾点/亮点/模糊点
反思:  「这个维度的考察够了吗？」「候选人的弱点是什么？」「该深挖还是跳过？」
决策:  • 追问 (在同一维度深挖)  → 返回 QUESTIONING
       • 切换维度 (进入下一个pending维度) → 返回 QUESTIONING  
       • 结束面试 (覆盖率达标或题库耗尽) → 进入 REPORTING
```

状态机模型：`INIT → INTRODUCTION → QUESTIONING → EVALUATING → REFLECTING → (loop) → REPORTING → DONE`

### 4.2 面试官人格注入（场景B核心）

**挑战**：不同岗位需要不同风格的面试官，如何用Prompt实现差异化角色扮演？

**方案**：4步人格初始化

```
Step 1: 根据JD自动判断岗位类型和级别
Step 2: 生成面试官背景（姓名、职位、10年+大厂经验的虚构背景）
Step 3: 注入风格参数（语气/追问烈度/亲和度/专业度）
Step 4: 生成面试策略（重点维度、提问顺序、时间分配）
```

| 人格类型 | 语气风格 | 追问烈度 | 适用场景 |
|----------|----------|:--------:|----------|
| 严厉技术总监 | 直击要害，不容含糊 | 🔴 高 | 高级技术岗 |
| 亲和HR | 温和引导，注重体验 | 🟢 低 | 初级/实习岗 |
| 挑剔业务负责人 | 结果导向，挖边界 | 🟡 中 | 产品/业务岗 |

### 4.3 LLM输出稳定性保障

**挑战**：LLM输出JSON格式不稳定，常见问题包括：Markdown代码块包裹、JSON截断、非法逗号、幻觉编造

**方案（三管齐下）**：

```
1. Prompt约束层: 
   - 明确要求「只输出JSON，不含Markdown」
   - 提供完整的JSON Schema示例
   - 防幻觉规则「只基于输入文本，不编造」

2. 客户端修复层:
   - 正则去除 ```json ... ``` 包裹
   - 正则修复尾部截断的JSON
   - 正则修复末尾多余逗号
   
3. 重试恢复层:
   - 指数退避 (1s → 2s → 4s)
   - 重试时注入「上次错误原因」帮助LLM纠正
   - 最多3次重试后降级处理
```

### 4.4 PDF解析鲁棒性

**挑战**：简历格式多样，包含文字型PDF、扫描件PDF、图片型PDF

| 级别 | 引擎 | 适用场景 | 失败处理 |
|:----:|------|----------|----------|
| 1 | PyMuPDF | 文字型PDF | 降级到2 |
| 2 | pdfplumber | PyMuPDF失败的文本型 | 降级到3 |
| 3 | RapidOCR | 扫描件/图片型 | 返回空或报错 |

### 4.5 TTS + 虚拟数字人同步（场景B）

**挑战**：面试官文本输出与TTS音频播放、Canvas虚拟主播口型动画需要同步

**方案**：文本输出 → TTS合成音频 → 播放；同步驱动Canvas mouthOpen参数实现口型动画。

### 4.6 题库抽取质量保障（新增）

**挑战**：从12+道题库中抽取N道（如8道）要保证维度覆盖均匀，不偏科

**方案**：分层抽样 + 保底机制。每个维度至少1题，剩余按随机补齐，剩余题库作为备选池供Agent追问时使用。

## 八、兼容性与注意事项

1. **端口统一**：合并后仅使用 8501 端口，场景A原FastAPI不再需要。
2. **nest-asyncio**：Streamlit内嵌Tornado事件循环，TTS异步调用需在import后立即apply。
3. **数字人模型**：荔枝离线数字人SDK体积大，保留为可选模块，默认Canvas虚拟主播。
4. **面试链接**：内存存储不变，后续可扩展数据库持久化。
5. **Prompt隔离**：场景A（招聘专家）和场景B（面试官）的System Prompt角色不同，各自独立。
6. **题库注入不丢失元信息**：抽取题目时完整保留 question/tags/hints/answer/difficulty，Agent评估时可对照参考答案。
