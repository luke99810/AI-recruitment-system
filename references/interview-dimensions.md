## 二、四维出题模板

### 维度 1：简历深挖（权重 40%）

#### 1.1 技术选型追问模板

```
模板：「你在 [项目名] 中使用了 [技术A] 实现 [功能]，为什么选择 [技术A] 而不是 [技术B] 或 [技术C]？在什么场景下 [技术A] 更有优势？」

示例：「你在 DeepOffer 中使用了 GPT-4 做简历优化，为什么不用开源的 LLaMA 或 Qwen？如果考虑成本和隐私，你会怎么选择？」
```

#### 1.2 实现细节追问模板

```
模板：「你提到 [项目名] 中 [具体功能/模块] 的 [指标] 达到了 [数值]，请具体说说你做了什么来实现这个效果？」

示例：「你说 HSR-SeqModel 的调度准确率提升了 15%，具体用了什么序列建模方法？LSTM 和 Transformer 在该场景下的表现差异是什么？」
```

#### 1.3 难点攻克追问模板

```
模板：「在 [项目名] 中遇到的最大技术挑战是什么？你如何定位和解决的？如果让你重新面对这个挑战，你会用什么不同的方法？」

示例：「在 OpenJiuwen 的 Docker + Alpine 流水线搭建中，你遇到了什么坑？Alpine 的 musl libc 和 glibc 兼容性问题你怎么处理的？」
```

#### 1.4 性能优化追问模板

```
模板：「[项目名] 目前的性能瓶颈在哪里？你做过哪些优化尝试？效果如何？如果数据量/用户量翻 10 倍，现有方案还撑得住吗？」

示例：「DeepOffer 中简历分析的响应时间是多少？如果同时有 1000 个用户请求，系统会有什么问题？如何设计高并发方案？」
```

#### 1.5 复盘思考模板

```
模板：「如果让你重新设计 [项目名]，你会改变什么？现在回头看，有哪些设计决策是可以优化的？」

示例：「回顾你的电价预测论文 TADLE，如果有更多时间和资源，你会在哪些方面做进一步研究？现有方法有什么局限性？」
```

#### 3.2 各岗位必备技能盲区表

| 岗位 | 必检盲区 |
|------|----------|
| 算法工程师 | 机器学习理论、深度学习基础、Transformer、Python 进阶 |
| AI 研发 | 数据结构算法、系统设计、数据库、Linux、Python/Go |
| AI 产品 | 数据分析基础、SQL、产品思维、用户研究 |
| 测试开发 | Python、测试框架、Linux、CI/CD、SQL |

**CS 基础通用盲区检测题目池：**

- 数据结构：链表、栈、队列、树、图、哈希表的实现与应用
- 算法：排序、搜索、动态规划、贪心、DFS/BFS
- 操作系统：进程/线程、内存管理、文件系统、并发/锁
- 计算机网络：TCP/UDP、HTTP/HTTPS、DNS、CDN
- 数据库：索引(B+Tree)、事务、SQL 优化、NoSQL 选型

**ML/DL 通用盲区检测题目池：**

- 监督/无监督/强化学习区别
- 梯度下降及变体（SGD/Momentum/Adam）
- 损失函数（交叉熵/MSE/Hinge）
- 过拟合应对：正则化/Dropout/Early Stopping/数据增强
- 评估指标：精确率/召回率/F1/AUC/ROC
- Transformer 完整结构与自注意力机制

## 三、追问逻辑（面试官思维）

### 3.1 STAR 追问法

```
S (Situation)  → 「当时是什么背景？」
T (Task)       → 「你的具体任务/目标是什么？」
A (Action)     → 「你采取了什么行动？为什么？」
R (Result)     → 「结果如何？如何衡量的？」
```

### 3.2 深度追问法（5 Why）

```
模板：针对候选人的回答，连续追问「为什么」5 次，直到触及根本原因

示例：
Q: 「你们用什么激活函数？」 A: 「ReLU」
→ 「为什么用 ReLU 而不是 Sigmoid？」
→ 「你说梯度消失，那 ReLU 就没有梯度问题了吗？」
→ 「那 Dying ReLU 怎么解决？」
→ 「为什么 Leaky ReLU 能解决？」
→ 「几何角度解释一下为什么？」
```

### 3.3 广度追问法

```
模板：从一个技术点引出相邻领域问题

示例：
Q: 「你们的模型用了 KV Cache 优化推理」
→ 「KV Cache 的原理？」
→ 「除了 KV Cache，还有哪些推理优化技术？」
→ 「这些技术之间的适用场景有什么不同？」
→ 「如果显存不够，优先选哪个优化？为什么？」
```

## 四、评分标准参考

### 4.1 技术题评分（1-3 星）

| 星级 | 标准 | 示例 |
|:---:|------|------|
| ⭐ | 知道概念，能说出定义 | 「LoRA 是一种参数高效微调方法…」 |
| ⭐⭐ | 理解原理，能解释 why | 「LoRA 通过低秩分解近似权重更新，因为…权重矩阵是低秩的…」 |
| ⭐⭐⭐ | 能推导、能对比、能举一反三 | 「LoRA 的 loss 函数设计相比 Adapter 的优势是…r=8 的选择需要考虑…在 13B 模型上验证…」 |

### 4.2 行为题评分（1-3 星）

| 星级 | 标准 | 示例 |
|:---:|------|------|
| ⭐ | 简单描述事件 | 「我们团队有过分歧，后来解决了」 |
| ⭐⭐ | 用 STAR 框架说明，有细节 | 「在 XX 项目中，因为技术选型我和后端有分歧…我通过做 benchmark 对比说服了他…」 |
| ⭐⭐⭐ | 有反思、有成长、有方法论 | 「那次冲突让我意识到…之后我养成了提前写 Design Doc 对齐方案的习惯…」 |

### 4.3 面试官挂人常见信号

- 简历写的技术一问三不知 → **直接挂**
- 项目回答模糊，没有具体细节 → **减分严重**
- 只会调用 API，不理解底层原理 → **看岗位，研发岗挂**
- 面对追问答非所问，不直面问题 → **减分**
- 态度消极，不愿承认不知道 → **减分**

## 五、题目难度分级

| 等级 | 适用轮次 | 特征 | 占总题量 |
|:---:|----------|------|:---:|
| **L1 入门** | 一面 | 概念题、基础原理、简单代码 | 20% |
| **L2 进阶** | 一面/二面 | 原理深入、技术对比、中等代码 | 40% |
| **L3 困难** | 二面/三面 | 系统设计、方案推导、开放问题 | 30% |
| **L4 挑战** | 三面 | 前沿论文理解、Tradeoff 分析、创新思路 | 10% |
